Architecture

The internal staff cockpit for O4F's Phase 1 release — strategy review, order management, risk control, and historical analysis. One SPA, three deploy surfaces: web, Android TWA, iOS WKWebView. This doc defines the structure the wireframes are built on.

The product in one paragraph

Every morning, the screener (derivatives repo) and external signal sources (Trendlyne PCR, short buildup, big-player flows, block deals) emit a list of FnO candidates. Rakesh currently narrows this to ~5 stocks/day and posts to Telegram. The cockpit replaces Telegram as the operational layer: candidates land in /today with their RMS-derived risk envelope; auto-qualifying low-risk candidates execute themselves via the OMS, high-risk candidates wait for admin approval, and the kill switch can veto everything in flight. Filled orders, PnL, and per-strategy attribution roll up into /history.

Roles & permissions

Three roles. Permission is additive — admin inherits user, super-admin inherits admin. All routes auth-gated; no implicit assumptions about read-only access: even viewing requires a session.

Why super-admin holds disarm but not arm: "kill the trading day" is reversible only by a senior. Anyone with admin can stop trades; only super-admin can resume after a kill. Prevents social-pressure resume mid-incident.

Critical flows

Flow 1 — Candidate → execution (hybrid)

[06:00 IST] screener (derivatives) + Trendlyne signals emit batch │ ▼ [ /today ] candidate lands; RMS evaluates risk envelope: SL, target, position size, notional value │ ├──── notional < ₹AUTO_MAX ─────────┐ │ │ │ ▼ │ status = AUTO_QUEUED │ countdown = 30s (admin can pause) │ │ if not paused │ ▼ │ status = APPROVED → OMS.PlaceOrder │ └──── notional ≥ ₹AUTO_MAX ─────────┐ │ ▼ status = NEEDS_APPROVAL waits indefinitely │ admin clicks Approve ▼ status = APPROVED → OMS.PlaceOrder │ ▼ status = WORKING (in OMS) │ WebSocket push (Zerodha) │ or polling (AngelOne/Dhan) ▼ status = FILLED / CANCELLED / REJECTED │ ▼ /history rolls up at EOD

Flow 2 — Kill switch

[ admin clicks kill ] │ ▼ choose scope: ⊙ all brokers ○ this broker ○ symbol = ___ │ ▼ impact preview shows: N candidates will be cancelled, M open orders will be cancelled │ ▼ type "KILL" + 3-second hold to confirm │ ▼ 1. OMS.SetSystemHealth(false) ← stops new orders system-wide 2. for each open order: OMS.CancelOrder(tag) 3. for each AUTO_QUEUED candidate: mark CANCELLED 4. emit kill event to audit log + all connected SSE clients │ ▼ [ red banner shown system-wide ] [ all WRITE actions disabled site-wide except super-admin "Disarm" ] │ ▼ super-admin clicks Disarm → status returns to normal (audit log entry: who, when, why-note required)

Flow 3 — RMS rule daily toggle

[ /rms ] rule "MAX_INTRADAY_LOSS_PCT = 2%" [today: ON | OFF ] rule "PER_SYMBOL_NOTIONAL_CAP = ₹100k" [today: ON | OFF ] rule "AUTO_EXECUTE_THRESHOLD = ₹50k" [today: ON | OFF ] ... │ ▼ admin toggles → state persists in DB with date=today │ ▼ RMS engine re-reads rules at evaluation time (no daemon restart) │ ▼ candidates landing AFTER toggle are evaluated under new rules; those already AUTO_QUEUED keep their original envelope

URL conventions

SPA routes

TanStack Router nested layout. /app wraps a RequireAuth boundary; /app/admin/* wraps a RequireRole<'super'>.

Deep links work everywhere (e.g. share /app/orders/ORD-abc in Slack).

API routes (BFF, proposed)

Versioned: /api/v1/.... JSON only. Server-Sent Events at /api/v1/stream with topic filters in query string.

Auth: short-lived JWT in Authorization header; refresh via httpOnly cookie.

Deploy surfaces

SurfaceHow it rendersConstraints to honor
Web (o4f.co.in/app) Vite SPA, public-static deploy None beyond same-origin auth
Android (TWA) Chrome Custom Tabs wraps the live web URL Must be HTTPS, must declare a digital asset link, must work offline-tolerant (network errors render gracefully)
iOS (WKWebView shell) Native shell loads the live web URL Must avoid window.alert (renders ugly), must size for safe-area insets, no third-party cookies allowed
Design rule we'll enforce in wireframes: Every screen must work in a 360×640 viewport. Bottom-nav for mobile (5 tabs max). Desktop layout is an enhancement, not a redesign.

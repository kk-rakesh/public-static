①
Top-right kill switch is always visible. Same position in every screen, every breakpoint. It's the single most important interactive element in the app — it must be muscle-memory reachable. Hover on this button reveals last-armed timestamp and impact summary; click takes you to /killswitch (not an instant kill — there's a confirmation flow on that screen).
②
Connection indicator shows SSE latency. Green dot when last server-push was <5s ago, amber when >5s, red when disconnected. The visible latency number (42ms) is the round-trip from the last heartbeat — it reassures the trader that what they're seeing is genuinely live, not a stale snapshot.
③
Stats strip is glanceable, not interactive. Numbers update via SSE. The Auto-firing stat turns amber when at least one candidate is in countdown — that's the "you have N seconds to intervene" signal. Click any stat to filter the list below.
④
Candidate card has three zones at desktop width: signal context (left — what the strategy thinks), risk envelope (middle — what RMS computed), execution state & actions (right — what the admin can do). This left-to-right ordering matches the reading flow: "what is it? · is it safe? · what now?"
⑤
Auto-queued candidates show a countdown timer. 30 seconds is the default grace period — long enough to react if the signal looks wrong, short enough not to be the whole strategy. Pausing extends to manual-approval status (no auto-fire). Cancelling kills the candidate entirely. Both are admin-only actions (regular users see read-only).
⑥
Needs-approval cards use a warm border (amber tint) so they stand out in a scrollable list. When the queue is long, this is the visual cue: "these are the ones blocking on you." The "Skip" action moves them to a cancelled state with reason captured in audit log.
⑦
Working orders stay on this screen until they reach a terminal state (FILLED / CANCELLED / REJECTED). They show the broker badge (Zerodha), the OMS ordertag, partial-fill counter, and elapsed time. The full timeline (modifications, fills) lives on /orders/:tag.
⑧
Right rail surfaces "what's the broader state." Signal feed = recent activity (read-only stream). System health = is everything wired up correctly? RMS rules summary = what governs decisions right now. The right rail is the first thing collapsed on smaller screens — its content is reassurance, not action.
⑨
Sidebar groups by intent. "Trading" = today's operational screens. "Strategy" = configuration and analysis. Settings/Admin are deprioritized at the bottom. This grouping survives at tablet width (icons-only), and gets replaced by bottom-nav at mobile.

⑪
Mobile drops to a 3-stat strip (Cand / Auto / Wait) and a horizontally-scrolling filter chip row. The cards stack vertically and reorganize the envelope into a 2×2 grid (Entry+Notional on top, SL+Target on bottom) because those are the natural reading pairs.
⑫
Bottom nav has 5 tabs, "Kill" in the middle. Center position because thumb reach is best there and the action is intentionally not buried under "More." This is the TWA experience — the muscle-memory equivalent of the top-right kill button on desktop. Same end behavior: it routes to /killswitch with the confirmation flow, not an instant kill.
⑬
Action buttons go full-width on mobile. Easier to hit with a thumb, and they don't fight for attention with the status pill above. Approve uses the brand green; Skip/Cancel stay muted because we want the affirmative action to be the obvious one.
⑭
The right rail and sidebar are gone on mobile. Signal feed and system health move to dedicated routes (/feed, /status) reachable from "More". The principle: on mobile, every screen does one job. Status reassurance is a separate screen.
STATE MATRIX

Candidate card states

Status Visual When Admin actions Next states
⟳ AUTO_QUEUED Blue border, countdown timer in amber Notional < AUTO_THRESHOLD RMS rule Pause → NEEDS_APPROVAL · Cancel → CANCELLED → WORKING (at countdown end)
⚠ NEEDS_APPROVAL Amber border, no timer (indefinite) Notional ≥ threshold, OR manually flagged, OR paused from AUTO Approve → APPROVED · Skip → CANCELLED → APPROVED → WORKING
● WORKING Green border, broker badge, partial-fill counter OMS has placed the order, broker is matching View → /orders/:tag · Cancel (calls OMS) → FILLED · CANCELLED · REJECTED
✓ FILLED Green pill, faded Broker confirmed total fill View only (read-only) Terminal · moves to /history at EOD
○ CANCELLED Muted, struck-through symbol Admin skip · auto-cancel · kill-switch sweep View only Terminal
✕ REJECTED Red pill, error reason inline Broker rejected · RMS pre-check failed View reason · retry (new candidate) Terminal
⑮
State machine sketch. The wireframe enforces this contract: a candidate is in exactly one state at a time, transitions are admin-initiated or broker-confirmed (never both racing), and terminal states (FILLED / CANCELLED / REJECTED) are immutable. Kill-switch sweeps perform a batch CANCELLED transition on every non-terminal candidate.
